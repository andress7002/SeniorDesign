# PWRSTATION_FW

Bare-metal firmware for an STM32F103C8T6 (Blue Pill) power monitoring station. Reads voltage, current, power, temperature, energy, and charge from a Texas Instruments INA228 85V precision power monitor and displays real-time engineering values on a DIYmall 0.96" SSD1306 I2C OLED display.

---

## Hardware

| Component | Part | Notes |
|---|---|---|
| Microcontroller | STM32F103C8T6 (Blue Pill) | 8 MHz HSI, no PLL |
| Power monitor IC | Texas Instruments INA228 | I2C address 0x40 (A0=A1=GND) |
| Display | DIYmall 0.96" SSD1306 128×64 OLED | I2C address 0x3C |
| Programmer | ST-Link V2 clone | SWD interface |
| Shunt resistor | 1 mΩ | Rated for max expected current |
| Pull-up resistors | 2.2 kΩ × 2 | SDA and SCL to 3.3V rail |

### Pin assignments

| Blue Pill pin | Function | Connected to |
|---|---|---|
| PB6 | I2C1 SCL | INA228 SCL, OLED SCL (shared bus) |
| PB7 | I2C1 SDA | INA228 SDA, OLED SDA (shared bus) |
| PC13 | Onboard LED (active-LOW) | Fault indicator / boot blink |
| PA13 | SWDIO | ST-Link SWDIO |
| PA14 | SWCLK | ST-Link SWCLK |

### INA228 wiring (high-side sensing)

```
Battery+ ──── VBUS
Battery+ ──── IN+  ──── [Rshunt 1mΩ] ──── IN- ──── Load
INA228 VS ─── 3.3V (Blue Pill)
INA228 GND ── GND
INA228 A0 ─── GND  (address bit 0)
INA228 A1 ─── GND  (address bit 1)
INA228 ALERT ─ NC  (not connected)
```

### I2C bus

Both devices share I2C1 on PB6/PB7. Pull-up resistors must connect from each line to the **3.3V rail** — not 5V. 2.2 kΩ gives a calculated rise time of ~186 ns, comfortably within the 300 ns Fast-mode budget.

## Software architecture

The firmware is structured as a non-blocking cooperative scheduler driven by `HAL_GetTick()` (1 ms Systick). There is no RTOS.

### Key files explained

**`main.c`** is the application layer. It contains three logical sections:

- `app_init()` — runs once at boot. Initialises the OLED and INA228 over I2C, verifies the INA228 silicon ID, configures the ADC mode and averaging, and resets the energy/charge accumulators. Uses PC13 LED blink codes to signal fault conditions without needing a serial monitor.
- `app_fast_task()` — called every 500 ms by the scheduler. Reads VBUS voltage, shunt voltage, current, power, and die temperature from the INA228 and writes them to OLED rows 0–3.
- `app_slow_task()` — called every 5000 ms. Reads the 40-bit energy and charge accumulators from the INA228 and writes them to OLED rows 4–5.

**`ina228.h` / `ina228.c`** implement the INA228 driver as a self-contained module. The public API covers initialisation, ADC configuration, accumulator reset, and all engineering-unit reads (V, I, P, T, E, Q). All I2C transactions use a finite 500 ms timeout (`INA228_I2C_TIMEOUT_MS`) so the firmware never hangs when the device is absent. A `_i2c_reset()` helper recovers the STM32F1 I2C peripheral from the hardware BUSY lockup bug after any timeout.

**`ina228_regs.h`** defines every INA228 register address and bit-field constant with the `INA228_REG_` prefix to avoid name collisions. Constants cover the full CONFIG, ADC_CONFIG, DIAG_ALRT register maps and known silicon ID values.

**`oled.h` / `oled.c`** implement the SSD1306 OLED driver in two layers. Layer 1 covers hardware primitives: `oled_init()` (full SSD1306 init sequence including charge pump), `oled_flush()` (single 1025-byte I2C burst using horizontal addressing mode), `oled_set_pixel()`, and line/rect drawing helpers. Layer 2 covers text rendering: `oled_print_str()` renders ASCII characters from an embedded 5×7 font stored in Flash, and `oled_printf_row()` formats a full 21-character row using `vsnprintf` and overwrites the entire row to prevent stale digit artifacts.

## Scheduler

The main loop is a simple non-blocking timer check with no blocking calls:

```c
while (1)
{
    if (ms_elapsed(s_t_fast) >= INTERVAL_FAST_MS)
    {
        s_t_fast = HAL_GetTick();
        app_fast_task();             // 500 ms: V, I, P, T
    }
    if (ms_elapsed(s_t_slow) >= INTERVAL_SLOW_MS)
    {
        s_t_slow = HAL_GetTick();
        app_slow_task();             // 5000 ms: Energy, Charge
    }
}
```

`ms_elapsed()` handles the 32-bit `HAL_GetTick()` rollover at ~49.7 days of uptime.

## Fault blink codes (PC13 LED)

| Pattern | Meaning |
|---|---|
| 3 fast (80 ms) | MCU alive — boot reached app_init |
| 7 medium (200 ms) | OLED not responding on I2C |
| 10 medium (200 ms) | INA228 not responding on I2C |
| 3 medium (200 ms) | INA228 found but silicon ID mismatch |

All faults are non-fatal. The firmware continues running in degraded mode after signalling the fault code.

## OLED display layout

```
Row 0   V:  12.345 V     bus voltage
Row 1   I:   3.456 A     load current
Row 2   P:  42.345 W     power
Row 3   T:   27.5 C      INA228 die temperature
Row 4   E: 1234.5 J      energy accumulator
Row 5   Q:  567.8 C      charge accumulator
Row 6   (reserved)
Row 7   (reserved)
```

Font: 5×7 pixel glyphs with 1-pixel column gap, 6 px per character, 21 characters × 8 rows on 128×64.

## INA228 configuration

| Parameter | Value | Notes |
|---|---|---|
| Shunt resistance | 1 mΩ | `INA228_SHUNT_OHMS` in main.c |
| Max expected current | 40 A | `INA228_MAX_AMPS` in main.c |
| Current LSB | ~76.3 nA/LSB | Computed as Imax / 2^19 |
| ADC mode | Continuous all | Shunt + bus + temperature |
| Conversion time | 1052 µs | Per channel |
| Averaging | 64 samples | ~67 ms total conversion cycle |
| I2C address | 0x40 | A0=GND, A1=GND |

To change shunt resistance or current range, update only the two defines in `main.c`. The driver recomputes `SHUNT_CAL`, `current_lsb`, and `power_lsb` automatically.

### Building

Clone the repository, open the folder in VS Code with the STM32CubeMX extension installed, and click **Build**. The extension handles CMake configuration automatically using `CMakePresets.json`.

To rebuild from scratch delete the `build/` directory and rebuild.

### Flashing

Connect the ST-Link V2 to the Blue Pill SWD header (SWDIO → PA13, SWCLK → PA14, 3.3V, GND), then click **Run and Debug** in VS Code. Verify BOOT0 jumper is set to 0 (GND) before flashing.

### Modifying CubeMX configuration

Open `PWRSTATION_FW.ioc` in STM32CubeMX. After making changes click **Generate Code**. CubeMX preserves all user code inside `/* USER CODE BEGIN */` / `/* USER CODE END */` markers. Always back up `main.c` before regenerating as a precaution.

## Adding this repository's driver files after CubeMX regeneration

CubeMX does not know about `Core/DriversExt/`. After regenerating, verify `CMakeLists.txt` still contains these entries:

```cmake
target_sources(${CMAKE_PROJECT_NAME} PRIVATE
    ...
    Core/DriversExt/ina228/ina228.c
    Core/DriversExt/oled/oled.c
)

target_include_directories(${CMAKE_PROJECT_NAME} PRIVATE
    ...
    Core/DriversExt/ina228
    Core/DriversExt/oled
)
```

CubeMX will not remove these lines (they are outside its managed sections) but it may reformat the file — verify after every regeneration.
